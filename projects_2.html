Internet Fridge API.

This wasn't one of the biggest courseworks I did and wasn't actually worth that much for the course. It was definitely one of the ones I enjoyed the most though! It comprised of designing an API for a fridge that connects to the internet using a Python binding for SQLite.

There were a couple of main areas of focus for the project: database design and creativity. Database design looked at following the correct normalisation principles in creating relational databases, getting to the correct normal form etc. Creativity was about what features you could get into the API, and what you thought would make a cool fridge.

This was definitely a case of one of those projects that developed through the "Yeah, but wouldn't it be cool if it... " mentality. By the end the API could email you things that had just gone out of date, look up products by their barcode on an internet database, recommend you recipes based on what you had in the fridge and even connect to Facebook to automatically post statuses about what you were eating (handy for a lot of people I know...).

Gravity.

Gravity was a game designed for a piece of Java programming coursework. It uses Swing graphics and one of the Java audio libraries to create a fun (ish) game where you play a little ball that jumps over obstacles as they come onto screen. There's powerups as well which is obviously insanely exciting. A massive thanks also goes to Rupert Cole who offered his incredible talent to do the soundtrack and FX for the game which, in my opinion, is what makes the game complete.

Some things to note about it are that the ball is animated under gravity, the need for threading and the OO design. The ball can switch between travelling along the top of the screen and the bottom, and the idea was to try and program that cleverly enough to add an extra dimension to the game, with minimal work.

Something I really wanted to try and take away from this project was the use of threading in programs. I had previously done some programming in C, so was used to a very procedural way of thinking, not a lot of things happening concurrently. Marrying into that is the concept of Object Orientated design, so thinking about classes having responsibilities rather than breaking a program down into linear steps.
